# Техническое задание — LawCRM (итоговая версия)

## 1. Общая архитектура

| Компонент | Технология |
|---|---|
| Бэкенд | Python 3.10+, FastAPI |
| База данных | SQLite (dev) / PostgreSQL (prod) |
| ORM | SQLAlchemy 2.0 |
| Веб-фронтенд | React 18 + TypeScript + Vite (уже реализован, лучше Jinja2) |
| iOS | Swift + SwiftUI |
| Android | Kotlin + Jetpack Compose |
| Аутентификация | JWT (мобильные + веб) |
| Пуш-уведомления | Firebase Cloud Messaging (FCM) |
| Фоновые задачи | APScheduler |

---

## 2. Роли пользователей

| Роль | Описание |
|---|---|
| admin | Полный доступ: все дела, все пользователи, логи, удаление |
| lawyer | Ведёт свои дела. Видит все дела (только чтение). Меняет только свои |
| viewer | Только чтение своих дел. Может добавлять вложения в свои дела |

## 4. Функциональные требования

### 4.1 Клиенты
- CRUD. Удаление только admin.
- Поиск по имени/телефону/ИНН.
- Conflict check: при создании дела проверять пересечения истец-ответчик с другими открытыми делами. Предупреждение без блокировки.
- Ведущие юристы клиента: до 3 юристов из справочника company_lawyers. Один помечается основным (is_primary). Управление через отдельные эндпоинты `/api/clients/{id}/lawyers`.

### 4.2 Дела
- CRUD. Удаление только admin.
- Выбор истца и ответчика из клиентов.
- Выбор ведущего юриста из company_lawyers.
- Автогенерация external_case_url: kad.arbitr.ru/Card/номер для арбитража.
- Полный список статусов: active/suspended/closed/won/lost/appeal_filed/cassation_filed/deadline_missed.

### 4.3 Стадии (инстанции)
- В карточке дела — блок "Движение по инстанциям".
- Список стадий с хронологией.
- Кнопки "Перейти в апелляцию", "Перейти в кассацию" — создаёт новую стадию, предыдущую помечает завершённой.
- Текущая активная стадия определяет hearing_date и judge в общем списке.
- Для каждой стадии API возвращает `stage_link` — ссылка на карточку дела в суде (вычисляется по `case_number_stage` и типу суда).

**Макет блока "Движение по инстанциям":**
```
┌────────────────────────────────────────────────────────────┐
│ ДВИЖЕНИЕ ПО ИНСТАНЦИЯМ                                    │
├────────────────────────────────────────────────────────────┤
│ ✅ ПЕРВАЯ ИНСТАНЦИЯ (завершена)                           │
│    Номер дела: А40-12345/2024 → [открыть в суде]          │
│    Судья: Иванова А.А. | Решение от 10.01.2025            │
├────────────────────────────────────────────────────────────┤
│ ⏳ АПЕЛЛЯЦИЯ (активная)                                   │
│    Номер дела: 09АП-6789/2025 → [открыть в суде]          │
│    Судья: Петров Б.В. | Заседание 20.03.2025 14:30        │
├────────────────────────────────────────────────────────────┤
│ [Перейти в кассацию]                                      │
└────────────────────────────────────────────────────────────┘
```
- Каждая завершённая стадия показывает ✅ и дату решения.
- Активная стадия показывает ⏳ и дату ближайшего заседания.
- Кнопка перехода показывается только для следующей инстанции.

### 4.4 Процессуальные сроки
- Поля decision_date, full_decision_date.
- Автоматический расчёт сроков:
  - Апелляция (арбитраж): 1 месяц от full_decision_date
  - Кассация (арбитраж): 2 месяца от даты вступления в силу
  - Надзор (ВС): 3 месяца от даты последнего акта
- Визуальный блок в карточке дела с таймером и цветовой индикацией.
- Уведомления: за 7, 3, 1 день, в день X, при пропуске срока.

### 4.5 Вложения
- Загрузка, скачивание, удаление.
- Права: admin/lawyer — любые дела, viewer — только свои.

### 4.6 Исполнительное производство
- Блок в карточке дела при статусе >= "Выдан ИЛ".
- CRUD для ИП.
- Кнопка "Проверить на сайте ФССП".

### 4.7 Дашборд
- Блок "Ближайшие заседания" (7 дней).
- Фильтр по инстанции: все/первая/апелляция/кассация/надзор.
- Блок "Просроченные сроки обжалования".

### 4.8 Логирование
- Каждое изменение (CREATE, UPDATE, DELETE, UPLOAD) фиксируется в audit_logs.
- Хранятся old_value и new_value (JSON).
- Логи неизменяемы, просмотр только admin.

### 4.9 Управление пользователями
- CRUD пользователей — только admin.
- CRUD company_lawyers — только admin.

## 6. Фоновые задачи (APScheduler)

| Интервал | Задача |
|---|---|
| Каждый час | Напоминания о заседаниях (за 3 дня, 1 день, 1 час) |
| Каждые 30 мин | Напоминания о сроках обжалования (за 7, 3, 1 день, пропуск) |
| Каждую ночь 00:00 | Пересчёт просроченных сроков, обновление статусов |

## 7. Мобильные приложения

### Общие требования
- iOS 15+, Android 8+ (API 26)
- Тёмная тема
- Офлайн-режим (SwiftData/Room)
- Биометрия (Face ID / Touch ID / Fingerprint)
- Пуш-уведомления FCM

### Экраны
- Вход (логин/пароль, биометрия)
- Дашборд (заседания, pull-to-refresh)
- Список дел (фильтры, поиск)
- Карточка дела (вкладки: Основное / Стадии / Вложения / ИП)
- Создание/редактирование дела
- Календарь (месяц/неделя)
- Профиль (настройки, смена пароля, выход)

### Специальные возможности
- Фотографировать документ и прикреплять к делу
- Позвонить клиенту (набор номера)
- Отправить email
- Открыть внешнюю ссылку в браузере
- Экспорт заседания в системный календарь

---

## 8. Критерии приёмки

1. Все CRUD-операции работают
2. Роли соблюдаются: admin видит всё, lawyer — только свои дела (чтение чужих)
3. Стадии создаются, переключаются, отображаются в карточке
4. Сроки обжалования рассчитываются автоматически, уведомления приходят
5. Дашборд: фильтр по инстанциям + блок просроченных сроков
6. Вложения загружаются, скачиваются, удаляются
7. Логи пишутся, неизменяемы, доступны admin
8. iOS/Android: подключается к API, работает офлайн, пуши приходят
